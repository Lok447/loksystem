整体介绍

LokSystem 不是单纯的聊天窗口，而是围绕“真实任务推进”设计的 AI 协作工作台。

本地 Agent 工作台

直接接入 Lok CLI、Hermes、Codex 等后端，在桌面端完成会话、文件读写和产物沉淀。

多成员 Team Mode

Leader 可以派发任务、回收结果、处理失败成员，并把未完成任务重新收回主线。

WebUI 与远程触达

同一 workspace 既能在桌面端查看，也能通过 WebUI 远程同步，让跨端验收更顺手。

核心能力

围绕发版、交付、回归这类高频场景，LokSystem 提供稳定的主链路能力组合。

桌面端打包与发布

支持 Windows x64 打包验证、安装包命名复查、About 页与静态官网页联动验收。

附件与文件链路

支持“新建会话 → 上传附件 → Lok CLI 读取 → 生成 workspace 文件 → 桌面端和 WebUI 可见”的主链路闭环。

回归与验收脚本

支持发版 smoke checklist、Team Mode 用户验收脚本和固定回归清单，减少临门一脚返工。

使用场景

当任务需要跨文件、跨 Agent、跨终端推进时，LokSystem 的优势会更明显。

稳定发版链路

从构建、打包、安装包验证，到静态官网页、版本号和升级说明，一次性收口。

Leader 协同推进

让 Leader 管理多名成员、重新分派失败任务、追踪主线结果并汇总输出。

Office 原生产出

把模型输出直接变成可交付的 Word、Excel、PPT、PDF，而不是停留在对话记录里。

快速开始

第一次使用 LokSystem 时，建议先跑通下面 4 步，确认桌面端、Agent、workspace 与远程页都工作正常。

Step 01 安装并启动

确认桌面端可启动，About 页品牌、图标、启动页与默认语言正常。

Step 02 连通主 Agent

使用 Lok CLI 或其他 ACP 后端创建会话，验证附件上传和 workspace 文件生成。

Step 03 验证 Team Mode

让 Leader 分配任务给两个成员，观察一个完成、一个失败时的任务回收与结果汇总。

Step 04 核对发版产物

运行打包命令并复查安装包命名、官网页跳转、升级说明和回归清单。

常见问题

以下问题围绕桌面端主链路、Team Mode 与离线打开场景整理。

这页离线能看什么？

站内锚点和本地图片完全离线可用；GitHub 仓库和 Releases 链接需要联网后才能访问。

LokSystem 更适合个人还是团队？

个人可以把它当成本地 AI 工作台，团队则可以把它扩展成 Leader + 成员的多 Agent 协作体系。

Team Mode 失败成员会怎样处理？

失败成员会被标记为 failed，Leader 会收到失败说明，同时未完成任务会尽量回收到主线，便于重新分派。

哪里查看最新安装包？

本地页主要用于说明与验收；真正的安装包和版本记录请以 GitHub Releases 为准。
